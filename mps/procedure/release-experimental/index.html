Memory Pool System Project

Memory Pool System Experimental-Release Procedure

Richard Kistruck, Ravenbrook Limited, 2008-01-07

1. Introduction

This is the procedure for making an experimental release of the Memory Pool System.

An experimental release is a quick way to make under-development MPS code available to external parties for testing. Such releases aren't part of a formal chain of supported releases and are only tested prior to shipping to the degree required by the experiment. See Justification, below.

Readership of this document is Ravenbrook development staff. (If you are a user of the MPS, and want to build object code from an mps-kit, please see the readme.txt file in the kit).

This document is not confidential.

2. Procedure

2.1. Prepare your modified MPS source

Experimental releases come from a task codeline and the released material is stored in that codeline: branch/YYYY-MM-DD/NAME/. (In the rest of this procedure, relative paths assume that this is your current directory).

2.2. Determine release name and origin

An experimental release's name is the Perforce changelevel of its origin, prefixed with "exp-".

The following procedure is cheap and usually works, despite the race condition.

  1. p4 changes -m 1, add 1: this is the expected changelevel NNNNNN for your experimental release.

  2. Edit code/version.c: set MPS_RELEASE to "exp-NNNNNN".

  3. Submit code/version.c.

  4. Inspect the output from the submission to ensure that you got the changelist number you expected. If you missed this for some reason, try p4 changes -m1 code/...#have, which will show the highest changelist number for the code that you have synced.

If someone else grabbed your intended changelevel, repeat until you win.

2.3. Make the release package

This example is for an experimental release to Configura.

  1. Check that you have no opened files in the code directory. If you have, either submit or revert them.

  2. Ensure that the SDK build environment is configured in this console window. This can be done using code/configure_sdk

  3. Clean directory, Sync to origin, Build:

    P4 sync code/...#none
    RMDIR/S/Q code
    P4 sync code/...@NNNNNN
    PUSHD code
    NTBUILD exp-NNNNNN
  4. Note that in previous incarnations of the MPS, the build was with earlier versions of MS Visual C installed in a specific location and the build sequence was: 
    \vc6path.bat
w3build.bat
\vc9vars.bat
w3vc9.bat

    The old build command procedures are retained in case we wish to reproduce these builds, but they should be considered obsolete and will be removed in due course.

  5. The build step should have produced two new subdirectories:
    w3i3mv
    Contains all the generated files from the build;
    exp-NNNNNN
    A release directory explicitly populated by the build command procedure with the artefacts we want to ship.
  6. Run the standard smoke tests for the release with the command: ../tool/test-runner

    Note that this is actually a source level test - it isn't explicitly testing the contents of the release directory because the current release manifest doesn't include a mechanism to do that.

    The test-runner program builds and executes a number of tests in each of the four build variants we produce. The results are written to a file called a1.txt in the current directory. Inspect this file to check that no unexpected errors occurred.

  7. Make a Zip Archive, in the local release directory. The example below uses the 7-ZIP utility, but a similar result can be achieved with the Info-ZIP utility, if preferred.

    if not exist ..\release\nul md ..\release
7z a ..\release\MMconfigura-exp-NNNNNN.ZIP .\exp-NNNNNN\*

    Also archive the test log file: 

    7z a ..\release\exp-NNNNNN_test_log.ZIP a1.txt

    release/ is a new "branch release directory", a peer of the /code directory. It is only present in branches from which we make experimental releases. Experimentally released files go here, for example: //info.ravenbrook.com/project/mps/branch/2007-04-18/diag/release/MMconfigura-exp-162399.zip.

  8. Add to perforce: 

    
  p4 add -t binary ..\release\MMconfigura-exp-162399.ZIP
  p4 add -t binary ..\release\exp-162399_test_log.ZIP
  p4 submit

    or equivalent.

2.4. Test and Publish

  1. Test as appropriate. For example, check that the ZIP file contents appear similar in number and size to a previous release that's known to be sound.
  2. Update the release list in //info.ravenbrook.com/project/mps/release/index.html with details of the release you are making.
  3. If necessary, make the webserver allow access to the release: see http://info.ravenbrook.com/procedure/webserver/ for details of what's involved.
  4. Send mail, including your (informal) release notes, and a link to the release on www.ravenbrook.com...zip (not info.ravenbrook.com!).

3. Justification and Design

An experimental release is a cheap and quick way to make under-development MPS code available to external parties (eg. customers), typically to let them evaluate new functionality.

This procedure was designed by simplifying and removing steps from the formal release-build procedure. For design justification, see http://info.ravenbrook.com/mail/2007/05/23/17-34-46/0.txt and http://info.ravenbrook.com/mail/2007/05/24/12-40-54/0.txt.
