Memory Pool System Project

Diagnostic output

This document contains a guide to MPS diagnostic output, followed by the historical initial design. References, History, Copyright and License are at the end.

Guide

Readership: any MPS developer. Not confidential.

Introduction

Diagnostic output is information emitted by the MPS for the purpose of helping MPS programmers and client-code programmers.

It is not intended to be end-user readable (or visible). It is not intended to be machine-parseable. It is not intended to be stable from one release to the next. It may be modified or removed at any time.

The steps in producing diagnostic output are:

  1. calculate, store, and propagate data;
  2. collate, synthesise, and format into a human-useful diagnostic message;
  3. control output of messages;
  4. use a channel to get the message out.

Note:

  • Keep the logic used to construct the human-useful message inside the MPS. This means that external utilities that display the messages do not need to understand, or keep in sync with, the details of what's going inside the MPS.

How to get diagnostic output

To run the MPS and get diagnostic output from it:

  • Use a variety with diagnostic output facilities compiled-in. Currently, that means variety.di. See config.h.
  • Check that the diagnostic output you require is generated, by looking in MPS source for invocations of the appropriate macro (eg. DIAG_WRITEF()). Currently, there is no way to control output of diagnostic messages: all diagnostics produced will be sent to the channel.
  • Run the MPS and client in an environment that supports the channel used (eg. at a command-line if using WriteF).

Channels

Example channels are:

  • printf;
  • WriteF;
  • a new type (yet to be defined) of mps_message;
  • squirt them into the telemetry-log-events system;
  • telnet.

Currently, only printf and WriteF are supported. See the DIAG_WITH_ macros in mpm.h.

Note: it is unfortunate that choice of channel may (for some channels) also dictate the form of the code that synthesises the message. (For example, WriteF-style parameter-expansion is not possible when using the printf channel, because there is no way to get WriteF to produce its output into a string). This is just a technical hitch; logically, the code that synthesises a diagnostic message should not care which channel will be used to transmit it out of the MPS.

Initial Design

RHSK 2007-06-28

It must be possible to add, modify, or remove diagnostic output in diagnostic varieties, without a substantial secondary effect on non-diagnostic varieties. It is not a requirement that there be zero effect.

This means:

  • for actual output, you must use DIAG macros;
  • for propagating and synthesising data, use appropriate DIAG macros where it is reasonably clean, such as for local variables and statements used only for diagnostic output;
  • where it is clearer and without substantial secondary effect to include code (such as additional function parameters) in all varieties, that's okay.
